# 00 目标与产品规格

## 你要的“研究助理”是什么（非 RSS 转发器）

**核心体验：低噪声 + 可追问 + 可回溯 + 可核验。**

- 低噪声：群里不会被刷屏（A 级快讯每日上限 10）
- 可追问：群里任何人都能 `/ask` 提问，得到引用与可执行建议
- 可回溯：日报全量覆盖，不因“没推送”而丢信息
- 可核验：A 级文章默认做最小核验，把“可信/不确定”讲清楚

---

## 输出层设计（强制分层）

### 1) A 级快讯（准实时）
触发条件：
- L1 评分为 A，并且当天 A 级推送未超过 10 条（超出进入日报，不实时推送）
- A 级会走 L2 核验后再推送（默认）

内容模板：
- 标题（保留英文标题）
- 3–5 条要点（中文，夹英文术语）
- “为什么重要：对 AI 产品经理的意义”（1–2 句）
- 重要性评分 + 置信度
- 链接

### 2) 每日 Digest（北京时间 07:00）
覆盖区间：**默认“自上次 digest 到本次 digest”**（更稳，兼容停机/延迟）

内容结构：
- 今日 Top 5（含产品解读）
- 主题分组（Agents/Eval/…）
- 趋势雷达：重复出现的方向（3–6 个）
- 建议：3 条可执行动作（实验/产品策略/工程落地）

### 3) 对话式问答（随时）
群里任何成员可用：
- `/ask <问题>`：问趋势、对比、落地建议
- `/topic agents`：只看某主题
- `/top 24h`：最近 24h 重要条目
- `/deepdive <id>`：对某篇做更深挖（可选）

---

## 关键非功能指标（工程验收用）

- **稳定性**：抓取/总结失败可重试，不会卡死 cron（cron 必须执行完退出）
- **幂等性**：同一篇文章不会重复入库、重复推送
- **可观测性**：每个阶段有日志与指标（成功/失败/耗时/队列长度）
- **成本可控**：对话与核验有 rate limit（防止群友刷爆账单）
- **安全**：博客内容视为不可信，防 prompt injection；管理命令仅管理员可用

---

## 技术选型（当前锁定）

- RSS 聚合：Miniflux（OPML 导入 + fetch-content 抓原文）
- 数据库：Postgres + pgvector（向量检索用）
- 工作流：Temporal
- Web 服务：FastAPI（Telegram webhook）
- LLM：DeepSeek-only，要求支持 JSON 输出
- 核验搜索：文内引用优先 + Tavily fallback
